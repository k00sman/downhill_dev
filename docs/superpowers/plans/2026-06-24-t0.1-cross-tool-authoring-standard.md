# Ticket 0.1 — Cross-tool skill/command authoring standard — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Define and prove one way to author a skill once and run it under both Claude Code and OpenAI Codex, by establishing the canonical-playbook + thin-per-tool-wrapper pattern and shipping one trivial reference entry.

**Architecture:** Pure docs + wrapper files. The real procedure lives once in `docs/playbooks/<name>.md`; each tool gets a thin `SKILL.md` wrapper that only defers to that playbook. Both tools use the **identical `SKILL.md` format** (frontmatter `name`+`description` + a one-line body) and differ only in directory: Claude Code reads `.claude/skills/`, Codex reads `.agents/skills/`. No gameplay code, no assets, no Unity build.

**Tech Stack:** Markdown only. "Tests" are structural/consistency checks (`ls`, `grep`, reading the files) plus following the wrapper→playbook chain by hand to prove it resolves. There is no headless way to drive Codex from here, so the live Codex slash-command smoke-test is handed to a human.

## Global Constraints

- **Cross-tool:** every reusable artifact must run under **both Claude Code and OpenAI Codex**. Never introduce a step only one tool can follow. `AGENTS.md` is the shared contract; `CLAUDE.md` is only a pointer.
- **Verified mechanism (Ticket 0.1 research, do not re-litigate):**
  - Claude Code project skills: `.claude/skills/<name>/SKILL.md`, frontmatter `name`+`description`, invoked `/<name>`. Claude Code does **not** read `.agents/skills/`.
  - Codex project skills: `.agents/skills/<name>/SKILL.md`, scanned from CWD up to repo root, invoked via `/skills` or implicitly by `description`. Codex custom prompts (`~/.codex/prompts/`, `/prompts:<name>`) are **home-only and deprecated** — do not use them.
  - Both tools follow a `SKILL.md` body that says "read and follow another file," so thin deferral works.
- **Unify on skills:** no `.claude/commands/` entries for cross-tool work (Claude treats commands ≡ skills; Codex has no repo-level command primitive). A skill is slash-invocable in Claude as `/<name>`.
- **`<name>` is kebab-case** and identical across the playbook file (`<name>.md`), the Claude wrapper dir (`<name>/`), and the Codex wrapper dir (`<name>/`).
- **No git operations.** The human handles all version control. This plan never stages, commits, or pushes — it ends by reporting changed files.
- **Do NOT touch** the linter (`tools/lint/`, `scripts/lint.*`, `.githooks/`, `.editorconfig`, `.gitignore`) or any gameplay code. No codemap regeneration is required (no `Assets/Scripts` change).

````markdown
# Cross-tool playbooks

This repo is driven by **both Claude Code and OpenAI Codex**. Anything reusable —
a skill, generator, command, or template — is authored **once, tool-neutrally**,
as a *playbook* in this directory, then exposed to each tool through a thin
wrapper. This file is the standard (Ticket 0.1); `AGENTS.md` carries the
one-paragraph summary, the detail lives here.

## The pattern

Three parts:

1. **Canonical playbook — `docs/playbooks/<name>.md`.** The real procedure,
   checklist, or template. Written in terms of *actions* ("read a file", "create
   a file", "run a command"), never one tool's API or tool-call syntax. Single
   source of truth — all logic lives here.
2. **Per-tool wrappers.** A thin skill per tool whose body only defers to the
   playbook. Both tools use the **identical `SKILL.md` format** — YAML
   frontmatter (`name`, `description`) plus a one-line body — and differ *only*
   in which directory the file lives in:

   | Tool | Wrapper path | Invoked as |
   |------|--------------|-----------|
   | Claude Code | `.claude/skills/<name>/SKILL.md` | `/<name>` |
   | OpenAI Codex | `.agents/skills/<name>/SKILL.md` | `/skills` → pick `<name>`, or implicitly by description |
3. **Contract — `AGENTS.md`.** The "Agent skills & commands (cross-tool)" section
   states the rule so every session (and human) knows playbooks are the source of
   truth and where wrappers live.

```
            docs/playbooks/<name>.md          ← procedure (write once)
              ▲                  ▲
              │ defers           │ defers
   .claude/skills/<name>/   .agents/skills/<name>/
        SKILL.md (Claude)        SKILL.md (Codex)
```

## Why skills for both (not custom prompts or commands)

- **Codex:** repo-committed, shareable reusable instructions are **skills**,
  discovered by scanning `.agents/skills/` from the working directory up to the
  repo root. The older custom-prompt mechanism (`~/.codex/prompts/<name>.md`,
  invoked `/prompts:<name>`) is **home-directory-only and deprecated** — not
  shared through the repo, so we don't use it.
- **Claude Code:** project skills are discovered only under `.claude/skills/` (it
  does **not** read `.agents/skills/`). Slash commands
  (`.claude/commands/<name>.md`) are now equivalent to skills — both produce
  `/<name>` — so we standardize on skills for one consistent pattern.

Verified against current docs in Ticket 0.1: the design spec's candidate
`.codex/prompts/` path was the deprecated mechanism; skills are the correct one.

## Wrapper boilerplate

Both wrapper files are byte-identical apart from their directory. Body = defer to
the playbook; nothing else.

`.claude/skills/<name>/SKILL.md` **and** `.agents/skills/<name>/SKILL.md`:

```markdown
---
name: <name>
description: <one line — what it does and when to use it. Codex uses this to
  decide implicit invocation, so make it specific.>
---

Follow the playbook at `docs/playbooks/<name>.md`: read it and do exactly what it
says.
```

## Conventions

- **`<name>` is kebab-case** and identical across all three files: playbook
  `<name>.md`, and each wrapper directory `<name>/`.
- **Keep wrappers thin.** No procedure text in a wrapper — if you're tempted to
  add a step to a `SKILL.md`, it belongs in the playbook. Duplicated logic is the
  one thing this pattern exists to prevent.
- **Write playbooks in actions, not tool calls.** "Create the file at X", not
  "use the Write tool". Each tool maps the action to its own toolset.
- **Tool-specific behavior is forbidden.** Never add a step only one tool can
  follow. If a step needs a capability the other tool lacks, redesign it.

## Adding a new cross-tool entry

1. Write `docs/playbooks/<name>.md` — the full procedure, tool-neutral.
2. Create `.claude/skills/<name>/SKILL.md` with the boilerplate above.
3. Create `.agents/skills/<name>/SKILL.md` with the same boilerplate.
4. If it's a standing workflow worth advertising, mention it in `AGENTS.md`.
5. Verify (below).

## Verifying an entry

- **Claude Code:** run `/<name>` (reload the session if the skill was just
  added). Confirm it reads the playbook and follows it.
- **Codex:** run it via `/skills` (or let description-based implicit invocation
  pick it). Confirm it resolves the same playbook.
- **Cross-tool limit:** one tool can't drive the other. As with the Unity Test
  Runner, when you can't run a tool yourself, **ask a human to run the other
  tool's side** and report — don't claim both verified from one tool.

## Reference entry

`example-cross-tool-check` is the trivial reference implementation of this
pattern — a playbook that does no real work, only confirms each tool resolved and
followed it. Copy it as the starting point for new entries.
````

````markdown
# Playbook: example-cross-tool-check

**Purpose:** A trivial reference playbook that proves the cross-tool authoring
pattern works — that both Claude Code and OpenAI Codex resolve and follow this
same file through their thin wrappers. It performs no real work; it only reports.

## Steps

When this playbook is invoked, do exactly the following, then stop:

1. Confirm you read this file by restating its purpose in one sentence.
2. Report which tool you are running under (Claude Code or Codex) and the wrapper
   path that invoked you.
3. Restate the cross-tool rule in one line: *the procedure lives once in
   `docs/playbooks/<name>.md`; each tool has only a thin `SKILL.md` wrapper that
   defers to it.*
4. Print the success line: `Cross-tool wiring verified for <tool>.`

## Expected output

A four-line report, one line per step. No files are created or modified.
````
